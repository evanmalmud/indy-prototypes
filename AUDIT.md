# AUDIT.md — measured, not asserted

Generated by `npm run audit` (`tools/audit.ts`). Every number here comes from an
exhaustive search over a pure `step(state, intent)` simulation — no level was
played, and no number was typed by hand.

**This document measures. It does not fix.** Levels flagged below are left exactly
as they are; repairing them is a separate piece of work.

## How to read this

- **Optimal** — shortest winning move count found by breadth-first search over canonical
  states. A trailing `?` means the search hit its budget and the number is a bound, not a fact.
- **Authored / Waste** — the length of the hand-authored route the test suite replays, and how
  many moves longer than optimal it is. Waste is not automatically bad: an authored route is a
  demonstration of the *intended* line, and the intended line is often deliberately scenic.
- **Distinct interactions** — `(tool, target)` pairs that actually fired along the optimal route.
  A pair is credited only when the step also changed the canonical state, so a tool aimed at
  something out of range or unaffordable is not counted as having interacted with it.
- **Declared but absent** — interactions the level's `requires:` line claims, which the optimal
  route never fires. This is the level lying about what it teaches.
- **Forced / declared** — the headline. For each declared interaction, that interaction is
  DELETED from the shared table and the level re-solved. The count is how many deletions made
  the level genuinely unsolvable. **A designed insight the puzzle does not force is a suggestion.**
- **Routes ≤ opt+2** — distinct winning move sequences within two moves of optimal, an ambiguity
  proxy. A trailing `+` means the count hit its cap. Routes differing only in the order of two
  irrelevant steps count twice, so read this comparatively, never as "alternate solutions".
- **Reachable states** — size of the reachable state graph, a dead-space proxy. A trailing `+`
  means the walk hit its node budget.

### Caveats that limit these numbers

1. **Deletion semantics.** Deleting a registry row makes `resolve()` return null, which every
   sim already treats as "no such pairing" — the ray passes through. For pseudo-interactions
   (traversal verbs, movement physics, world events) each sim consults the same disable flag at
   the point of occurrence. Two deletions have a judgement call baked in: `COLLAPSE+BOULDER`
   makes the floor hold under a stone rather than dropping it, and `TORCH+DARK` stops carried
   fire answering darkness while leaving brazier light intact.
2. **Pillar C's `requires:` is documented as describing the PAR ROUTE, not the set of necessary
   interactions** (`src/pillar-c/levels.ts`). An `AHA NOT FORCED` verdict there is therefore
   weaker evidence than the same verdict in A or B: pillar C is built on substitution, so the
   cheapest route being a subset of the par route is the design working, not failing.
3. **Pillar C cannot be lost and does not require carrying treasure out** — reaching the exit
   with empty hands always wins at score 0. So "cheapest route" is nearly always the empty-handed
   walk, and the cost-gradient column is really asking whether spending ever buys more than it costs.
4. **Budget-limited rows are marked, never rounded off.** Where a search did not complete, the
   verdict is `UNKNOWN` rather than a guess.
